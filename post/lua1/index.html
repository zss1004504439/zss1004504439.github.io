第一篇：Redis 分布式锁 vs Lua 原子脚本：适用场景、优缺点、到底该怎么选

前言

很多初学者容易混淆两个方案：Redis 分布式锁（set NX EX）与 Redis Lua 原子脚本。有人以为所有并发问题都能用分布式锁，有人不知道什么时候该换 Lua。本文一次性讲透：原理、区别、适用场景、选型标准，看完以后业务直接不会选错。

一、两种方案核心原理回顾

1. Redis 分布式锁（setNX EX）

伪代码示例（新手可直接参考，关键步骤带注释）：

// 抢锁：NX=只有key不存在时才能设置成功，EX=设置10秒过期时间（防止死锁）
// 锁key：lock:buy（自定义，建议按业务命名，如lock:goods:1001）
// 锁值：1（无实际意义，仅用于占位）
const lock = await redis.set('lock:buy', '1', 'NX', 'EX', 10);
// 抢锁失败：返回null，提示用户排队重试
if (!lock) return '排队中，请重试';

try {
  // 抢锁成功：执行业务逻辑（查库存、创建订单、扣库存等）
  // 此处业务逻辑执行期间，其他请求会因抢不到锁而排队
} finally {
  // 无论业务成功/失败，最终都要删除锁，释放资源（防止死锁）
  await redis.del('lock:buy');
}

核心逻辑：同一时间只允许一个请求执行业务，其他人全部排队等待，通过“互斥”保证数据安全。

2. Redis Lua 原子脚本

把「查库存 → 判断 → 扣库存」多条 Redis 命令，写到一段 Lua 脚本里。Redis 内部单线程串行执行整条脚本，中途不允许其他请求插队，天然具备原子性，从根源上防止超卖，无需手动控制互斥。

二、核心区别对比表

维度 分布式锁（setNX） Lua 原子脚本
并发能力 同一时间只能1人执行业务 有多少库存就支持多少并发
原子保障 靠锁互斥，锁住整个业务流程 Redis 单线程 + 脚本原子，命令级不插队
性能 低，串行排队等业务逻辑（耗时久） 极高，只在Redis内存微秒级执行（耗时短）
超卖风险 锁设计不好有风险（误删、过期、续命） 天然无超卖，原子执行无间隙
代码复杂度 高，需处理锁过期、锁续命、防误删等坑 低，脚本一次写完，无需关心锁相关逻辑
适用业务 唯一名额、抢单、抢优惠券、独占资源 普通商品库存、秒杀多库存、限流、扣余额
维护成本 高，坑多容易出线上bug，需持续维护锁逻辑 低，一次脚本通用所有商品，无需额外维护

三、分布式锁适合什么场景？

适用场景

资源只有 1 个，同一时刻只能有一个人操作（核心是“独占”）：

  • 抢唯一优惠券、抢限量1个的名额（如限量签名照）

  • 任务独占执行（定时任务防止多实例重复跑，如每日凌晨统计订单）

  • 单笔独占操作、不能多人同时进入业务（如用户提现审核）

致命缺点

  1. 多库存商品（库存10、100）不能用：明明有10个库存，也只能一个一个排队买，浪费并发，用户体验差。

  2. 坑非常多，新手极易踩雷：

  3. 锁超时业务没执行完：10秒锁过期，业务还在执行，其他请求抢锁成功，导致并发错乱

  4. 业务没走完锁被误删：比如A执行业务时，误删了B的锁，导致B的业务被打断

  5. 需要做锁续命、防删除误判：需额外写代码处理锁续命（业务超时自动续期），复杂度飙升

一句话：只有「独占唯一资源」才用分布式锁，多库存场景绝对不推荐。

四、Lua 脚本适合什么场景？

适用场景

有多库存、允许高并发、要防超卖（核心是“原子扣减”）：

  • 电商商品扣库存（如库存100件，允许100人同时下单）

  • 秒杀多件商品（如秒杀100件，支持100人并发抢购）

  • 用户扣余额、扣积分、扣次数（如用户积分扣减，需原子操作防多扣）

  • 接口限流、防重复提交（如IP限流，原子计数防并发错乱）

  • 批量原子操作、多条件判断扣减（如“库存&gt;0且用户未购买”才能扣减）

优势

  1. 不限制并发人数：有多少库存就支持多少人同时操作，并发拉满

  2. 执行速度极快：只锁住 Redis 命令执行阶段（微秒级），不阻塞业务逻辑，用户无感知

  3. 零额外坑：无需关心锁、释放锁、续命，脚本写完直接用

  4. 业内标准：电商、秒杀场景的最优解，各大公司均采用此方案

一句话：但凡多库存、高并发、要原子扣减，优先用 Lua，简单又安全。

五、经典业务选型结论

  • 场景1：商品库存 10件、100件 售卖 → 选 Lua 原子脚本（不要用分布式锁，浪费并发、用户排队体验差）

  • 场景2：秒杀只有 1 个名额、抢唯一奖品 → 选 Redis 分布式锁（用 Lua 没必要，本来就只能卖1个）

  • 场景3：微服务定时任务，防止多实例同时执行 → 分布式锁

  • 场景4：用户限流、防重复下单、扣积分余额 → Lua 脚本

六、常见误区纠正

  • 误区1：只要并发问题，都用分布式锁 → 错！分布式锁是业务级别串行，并发极低；Lua 是 Redis 命令级别原子，并发拉满，多库存场景用分布式锁会严重影响性能。

  • 误区2：Lua 也能当分布式锁用 → 可以，但没必要。Lua 擅长多并发原子扣减；分布式锁擅长独占资源互斥，用 Lua 实现互斥会浪费其高并发优势。

  • 误区3：Redis 多线程会不会破坏 Lua 原子性 → 不会。Redis 新版网络IO是多线程（提高网络传输效率），但命令执行依然是单线程串行，Lua 脚本作为一条命令，原子性不受影响。

七、最终总结

  1. 抢唯一、独占资源 → 用 Redis 分布式锁（核心：互斥）

  2. 多库存、高并发、扣库存/扣余额/限流 → 用 Lua 原子脚本（核心：原子）

  3. 普通电商商品下单、秒杀多件，Lua 是首选、最简单最稳，新手优先掌握

  4. 分布式锁不要滥用，多库存业务乱用会严重限制并发、影响用户体验，甚至引发性能问题

第二篇：为什么普通分步查库存 + 扣库存 一定会超卖

前言

很多新手写电商下单逻辑，习惯用直观的两步写法：1. 先从 Redis 查出当前库存；2. 前端/业务层判断库存 &gt; 0，再执行扣库存。本地单机测试没问题，一上高并发、多请求同时进来，立马超卖。本文讲透底层原因、时序流程、根本问题，帮新手避开这个高频坑。

一、错误代码示范（新手常写，必超卖）

// 错误写法：分步查库存、扣库存（高并发必超卖）
async function buyWrong(goodsId) {
  // 拼接库存key：格式为 goods:stock:商品ID，方便区分不同商品库存
  const key = `goods:stock:${goodsId}`;
  // 第一步：查库存（从Redis获取当前库存值）
  let stock = await redis.get(key);
  // 转成数字类型（Redis返回的是字符串，需手动转数字）
  stock = Number(stock);

  // 第二步：业务层判断（在Node.js代码中判断，非Redis内部判断）
  if (stock > 0) {
    // 第三步：扣库存（库存充足，执行扣减1操作）
    await redis.decr(key);
    // 创建订单...（后续业务逻辑，如写入数据库）
  }
}

逻辑看着没问题，本地单机测试（一次只有一个请求）也能正常运行，但高并发下必超卖，核心问题出在“分步操作”上。

二、超卖现场还原（直观理解为什么会超卖）

初始库存：1（商品只有1件），同时两个请求 A、B 并发进来（高并发场景常见）：

  1. 请求A：执行 get 命令，查到库存 = 1

  2. 请求B：同时执行 get 命令，也查到库存 = 1（两步操作有间隙，B趁机插队查库存）

  3. A 判断 1&gt;0，准备执行扣库存操作

  4. B 判断 1&gt;0，也准备执行扣库存操作

  5. A 先执行 decr 命令 → 库存变 0

  6. B 后执行 decr 命令 → 库存变 -1（库存为负，超卖）

结果：1件库存，卖出去2次，直接超卖，造成商家损失，这是高并发场景下的致命bug。

三、根本原因（3个核心，新手必懂）

1. 两次网络请求，中间有时间窗口

get 是一次 Redis 网络请求（从Node.js发送到Redis，再返回结果），decr 是第二次 Redis 网络请求。两次命令不连续，中间有空闲间隙（哪怕只有几毫秒），其他并发请求可以完美插队，查到相同的旧库存。

2. 业务代码层面做判断，不是 Redis 原子判断

库存判断放在了 Node.js 业务层，不是在 Redis 内部执行。Redis 无法保证“查库存”和“扣库存”的连续性，所有并发请求都能读到旧库存，进而全部通过判断、执行扣减。

3. Redis 命令执行虽单线程，但管不了你分步逻辑

Redis 内部命令是单线程串行执行的，但它只能保证“单个命令”的原子性，管不了你“先调 get、再隔一会调 decr”的分步逻辑，中间的间隙依然会被其他请求插队。

四、一句话总结坑点

- 先查后扣是两条独立命令，有间隙可插队；- 库存校验放在业务代码，不是 Redis 内部，无法保证原子性；- 高并发下大量请求同时读到相同旧库存，集体通过判断 → 超卖。

五、正确解决方案

不要在业务层拆分「查、判、扣」三个步骤，把整套逻辑交给 Redis Lua 脚本原子执行：查库存 → 判断 → 扣库存，Redis 内部一次性跑完，不允许任何请求插队，从根源上杜绝超卖。

第三篇：Redis + Lua 原子扣库存 完整可运行代码

前言

解决超卖最优方案：Redis Lua 原子脚本。利用 Redis 单线程特性，把查库存、判断、扣减合并成一条原子命令，天然防超卖，并发性能拉满，新手可直接复制到项目中使用，无需额外修改核心逻辑。

一、安装依赖

使用 Node.js 开发，需先安装 Redis 客户端 ioredis（轻量、易用，业内常用）：

npm i ioredis

二、完整可运行代码（带详细注释，新手易懂）

// 1. 引入并初始化 Redis 客户端（连接本地Redis，默认端口6379）
// 若Redis部署在远程服务器，需修改host、port，如需密码添加password字段
const Redis = require('ioredis');
const redis = new Redis({
  host: '127.0.0.1', // 本地Redis地址，远程请替换为服务器IP
  port: 6379         // Redis默认端口，未修改过则无需改动
});

// 2. Lua 原子扣库存脚本（核心逻辑，一次性执行查、判、扣，无间隙）
const stockLua = `
  -- 接收外部传入的库存key（从JS中通过KEYS[1]传入，对应goods:stock:商品ID）
  local stockKey = KEYS[1]
  -- 获取当前库存：redis.call('get', stockKey)查库存，不存在则默认0，转成数字类型
  local stock = tonumber(redis.call('get', stockKey) or 0)
  -- 判断库存是否充足（大于0则可扣减）
  if stock > 0 then
      -- 库存充足：执行扣减1操作（decr命令自动减1）
      redis.call('decr', stockKey)
      return 1  -- 返回1，告诉JS：扣库存成功，可以下单
  else
      -- 库存不足：不执行扣减，返回0
      return 0  -- 返回0，告诉JS：商品已售罄
  end
`;

/**
 * 3. 购买下单核心函数（对外提供的接口，可直接调用）
 * @param {string|number} goodsId 商品ID（区分不同商品，如1001、1002）
 * @returns {string} 下单结果（成功、售罄、失败）
 */
async function buy(goodsId) {
  // 拼接当前商品的库存key，格式：goods:stock:商品ID
  const stockKey = `goods:stock:${goodsId}`;

  // 执行Lua脚本：参数1=脚本内容，参数2=传入Lua的KEYS数量（此处1个），参数3=具体的KEY值
  const res = await redis.eval(stockLua, 1, stockKey);

  // 判断Lua返回结果：0=库存不足，直接返回售罄
  if (res === 0) {
    return '商品已售罄';
  }

  // 走到这里：Lua扣库存成功，开始执行业务逻辑（创建订单、写库等）
  try {
    // 此处写你的真实业务逻辑（根据自己的项目修改）
    // 1. 创建订单（生成订单号、关联商品和用户）
    // 2. 写入数据库（订单表、用户订单关联表等）
    // 3. 其他业务（扣减用户余额、发放优惠券、发送下单通知等）
    return '下单成功';
  } catch (err) {
    // 异常处理：业务执行失败（如数据库报错），必须回补库存
    // 防止“库存扣了，但订单没创建成功”，导致库存数据不一致
    await redis.incr(stockKey); // incr命令：库存加1，回补之前扣减的数量
    return '下单失败，库存已回补';
  }
}

// 4. 初始化商品库存示例（仅首次执行，给商品设置初始库存）
// 商品ID=1001，初始库存=10件（根据实际商品修改ID和库存数量）
// await redis.set('goods:stock:1001', 10);

// 5. 导出函数（供其他文件调用，如接口层、服务层）
module.exports = { buy };

三、核心代码解释（新手必看，理清逻辑）

1. redis.eval 作用（执行Lua脚本的核心方法）

eval 是 Redis 官方提供的、执行 Lua 脚本的专属方法，语法固定，新手直接套用即可：

redis.eval(脚本内容, KEYS数量, KEYS1)

示例中：脚本内容=stockLua（我们写的扣库存脚本），KEYS数量=1（只传入1个库存key），KEYS1=stockKey（当前商品的库存key）；Lua脚本中通过 KEYS[1] 接收这个key，实现灵活复用（适配所有商品）。

2. Lua 脚本逻辑（核心中的核心）

  1. 接收商品库存 key：通过 KEYS[1] 接收JS传入的库存key，适配不同商品，无需重复写脚本。

  2. 获取库存值：查不到库存时默认0（避免null报错），转成数字类型方便判断。

  3. 库存大于0则扣减 1，返回 1 代表可下单：decr命令自动扣减1，原子执行，无间隙。

  4. 无库存返回 0 代表售罄：不执行扣减，直接返回结果，避免超卖。

3. try/catch 回补库存（关键细节，防止数据不一致）

Lua 扣库存成功后，如果创建订单、写库等业务报错（如数据库宕机、网络异常），会出现“Redis 库存扣了，但订单没生成”的问题，导致库存数据不一致。此时通过 catch 捕获异常，用 incr 命令把库存加回去，保证“要么下单成功，要么库存退回”，数据最终一致。

四、使用方式（新手直接复制使用）

// 1. 先初始化库存（仅首次执行，后续无需重复）
// 商品ID=1001，库存=10件（根据实际需求修改）
await redis.set('goods:stock:1001', 10);

// 2. 调用下单函数（传入商品ID，获取下单结果）
let ret = await buy(1001);
// 3. 打印结果（可根据需求返回给前端）
console.log(ret); // 输出：下单成功 / 商品已售罄 / 下单失败，库存已回补

五、优势总结（为什么选这个方案）

  1. 原子执行：Redis 单线程串行执行 Lua，不会插队，永久防超卖，从根源解决问题。

  2. 高并发：有多少库存支持多少并发，不人为串行阻塞，用户体验好，性能拉满。

  3. 代码简单：不用写分布式锁、不用处理锁过期、锁续命，新手也能快速上手。

  4. 数据安全：业务异常自动回补库存，保证数据最终一致，避免少卖、超卖。

六、适用场景

  • 电商商品下单扣库存（核心场景）

  • 秒杀、抢购多件商品（高并发场景）

  • 用户积分、余额扣减（原子操作需求）
