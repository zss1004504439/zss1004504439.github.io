GET 与 POST 接口缓存机制

2025-06-20 · 2 分钟阅读
缓存

📄 关于 GET 与 POST 接口缓存机制的调研总结

一、 核心结论

  1. GET 接口： 具备全链路缓存能力，是 Web 性能优化的核心手段。
  2. POST 接口： 默认情况下几乎不具备浏览器和网络层级的缓存，其缓存通常局限于服务端内部（业务层）。
  3. 设计原则： 遵循“读用 GET，写用 POST”。若因参数过长导致查询接口必须使用 POST，需通过额外技术手段手动实现缓存。

二、 缓存覆盖层级对比

我们将缓存分为四个层级，对比两者在各层的原生支持情况：

缓存层级 GET 请求 (默认) POST 请求 (默认) 关键点
L1: 浏览器缓存 完美支持 不支持 POST 的非幂等性导致浏览器默认强制请求服务器
L2: CDN/边缘计算 完美支持 不支持 CDN 节点通常只根据 URL 缓存，不解析 POST Body
L3: 反向代理 (Nginx) 支持 不支持 缓存 Key 默认不包含 Body，POST 缓存需特殊配置
L4: 服务端 (Redis/内存) 广泛应用 部分应用 POST 需将 Body 进行 Hash 运算后作为缓存 Key

三、 为什么 POST 接口难以缓存？

  • 语义冲突： POST 在 HTTP 规范中被定义为“非幂等”操作（如新增、修改），缓存可能导致服务器状态更新被跳过。
  • Cache Key 复杂性： * GET 的唯一标识是 URL + QueryString。
    • POST 的标识隐藏在 Request Body 中。Body 格式多样（JSON, XML, 二进制），中间节点提取并计算其摘要的计算成本过高。
  • 安全性考量： POST 常用于传输敏感信息（密码、个人信息），缓存这些数据存在极大的泄露风险。

四、 针对“必须缓存 POST”场景的解决方案

在某些复杂查询（如参数超过 2KB 的搜索接口）必须使用 POST 时，可采用以下补救措施：

  1. 服务端 Hash 缓存（L4 层级）：
    • 实现： 在后端接收到请求后，对 JSON Body 进行 MD5 散列，结合用户 Token 作为 Redis Key。
    • 优点： 减少数据库压力，逻辑受控。
  2. Service Worker 拦截（L1 层级）：
    • 实现： 在前端利用 Service Worker 拦截 POST 请求，手动将结果存入 CacheStorage。
    • 优点： 实现“零延迟”响应，绕过浏览器限制。
  3. 网关层特殊配置（L3 层级）：
    • 实现： 配置 Nginx 利用 $request_body 计算散列作为缓存 Key。
    • 缺点： 增加网关性能开销，对 Body 顺序敏感（即使字段相同，顺序不同也会导致缓存失效）。

五、 汇报建议与最佳实践

  • 性能优先原则： 对于纯查询接口，优先进行参数瘦身并回归 GET 模式，以获得最优的 CDN 和浏览器加速效果。
  • 业务解耦： 如果必须使用 POST 且需高性能，建议在服务端使用 Redis 缓存结果，不要过度依赖浏览器或网络节点的默认行为。
  • 安全提示： 任何涉及敏感数据或会导致数据库状态变更的 POST 接口，严禁开启任何形式的缓存。
